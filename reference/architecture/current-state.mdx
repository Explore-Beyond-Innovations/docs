## Protocol contracts

Both chains carry the same contract shape â€” Soroban / Rust on
Stellar, Solidity / Foundry on Ethereum â€” with identical EIP-712
order hashing semantics so a single ZK proof can settle both
sides.

| Contract | Role |
|---|---|
| `AdManager` | Maker-side escrow. Holds liquidity ads. Maker calls `createAd`, `fundAd`, `lockForOrder`, `unlock`. |
| `OrderPortal` | Bridger-side escrow. Bridger calls `createOrder` to deposit, `unlock` to settle. |
| `MerkleManager` | Append-only Poseidon2 MMR. `MANAGER_ROLE` (held by `AdManager` and `OrderPortal`) appends order hashes. |
| `Verifier` | UltraHonk on BN254 with keccak Fiat-Shamir. Verifies the deposit proof on-chain. |
| `wNativeToken` | Wraps the chain's native asset (ETH, XLM) so bridging works through the same ERC-20-style interface. |

See [Smart Contracts](/reference/smart-contracts) for the function
signatures and the deployed testnet addresses.

## What works today

- **Two active bridge routes** on testnet: ETH (Sepolia) â†” wETH
  (Stellar) and XLM (Stellar) â†” wXLM (Sepolia). A ProofBridge
  test-token route (PB-EVM â†” PB-Stellar) is also live for
  integration testing.
- **The 12-step cross-chain flow** described in
  [How it works](/how-it-works) runs end-to-end against real
  testnet contracts.
- **Cross-chain digest invariant**: a 15-field `Order` struct
  hashes to byte-identical EIP-712 digests on Stellar and
  Ethereum, so one ZK proof settles both sides.
- **Append-only MMR** on each chain with Poseidon2 hashing â€”
  the same hash function used inside the Noir circuit, so the
  on-chain root is directly referenceable as a public input.
- **Auto-merchant bot** runs against a ProofBridge admin wallet on
  each chain, auto-locking incoming orders so testnet bridgers
  never hit a "no merchant answered" dead end.

## What's transitional

This is the part Phase 2 retires. Three pieces of scaffolding
live in the current architecture only because the trustless
end-state isn't fully shipped yet:

- **Pre-auth signing.** Every state-changing user call carries a
  manager-signed request hash. The relayer's manager key is
  whitelisted in `managers[]`, gating the call. A compromised
  manager key can authorize any user request, though it still
  cannot redirect funds without a valid ZK proof.
- **Single-key admin** on each chain. `setChain`,
  `setTokenRoute`, `setManager`, and role grants on
  `MerkleManager` all execute instantly with no delay or multisig.
- **Native-token wrapping** uses an internal `wNativeToken`
  contract via a sentinel address. The protocol-level claim of
  "native handled directly" is an aspirational simplification of
  what the contracts actually do today.

## Trust assumptions

| Assumption | Why it holds today | What changes later |
|---|---|---|
| Relayer cannot redirect funds | ZK proof verification gates every `unlock` | Pre-auth path retired in Phase 2 |
| Funds locked in escrow are recoverable only via valid proof | Nullifier + EIP-712 binding + MMR inclusion proof | Unchanged through all phases |
| Admin acts in good faith on `setTokenRoute`, `setChain` | Single-key + no on-chain check | Phase 2 adds route-commitment defense; Phase 4 adds 24h timelock + multisig |
| Manager key is honest about which user requests to sign | Trust scaffolding while BLS is being built | Phase 2 retires this entirely |
