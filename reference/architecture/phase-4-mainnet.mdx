Phase 4 is the production-ready architecture. Most of the heavy
contract work is already in Phases 1–3; Phase 4's job is to
harden the operational surface so the protocol can hold real
mainnet value.

## Architecture diagram

<img src="/reference/architecture/_diagrams/phase-4-mainnet.svg" alt="Phase 4 architecture — Mainnet end-state" />

Compared to [Phase 3](/reference/architecture/phase-3-agents-disputes):

- **Single-key admin** is replaced by an **Admin Multisig** on
  each chain. Admin operations flow through a new **Timelock**
  module with a 24-hour delay before execution.
- **`ArbiterRole`** moves from "= admin" to its own dedicated
  **Arbiter Multisig** (5-of-7, 3 team + 4 ecosystem signers).
- **Protocol fees** become live on `unlock` — 0.2% to the new
  **Fee Pool**, 0.3% retained as maker LP profit. Funded from
  the destination payout.
- **Per-route bond config** in `AdManager`: bond denomination,
  floor, and bps ratio set per route. USDC bonds for the launch
  route.
- A new off-chain **Reconciliation Listener** watches both
  chains and alerts on settlement discrepancies. Read-only — no
  on-chain authority.
- The **launch route** is `USDC ↔ USDC` only. Native-token
  routes wait for ecosystem partners to issue equivalents
  cross-chain.

## What changes at the contract layer

<CardGroup cols={2}>
  <Card title="Timelock module" icon="clock">
    24-hour propose → execute delay on every admin operation that
    can move funds, redirect flows, or change auth gating.
    Pause / unpause exempt for emergency response. Bootstrap mode
    lets initial deploy run instantly; `activate_timelock()` is
    the one-shot irreversible flag flip before multisig handoff.
  </Card>
  <Card title="Multisig admin" icon="users">
    Stellar account-level multi-sig (signer thresholds + weights
    on the admin's classic account) and Gnosis Safe on EVM.
    Same M-of-N composition both sides; ≥ 2 non-team signers as
    a hard launch gate.
  </Card>
  <Card title="ArbiterRole multisig" icon="scale-balanced">
    5-of-7 separate multisig (3 team + 4 ecosystem). Quorum of
    5 from a 3-team-4-ecosystem pool guarantees ≥ 2 non-team
    signers in any decision by construction. Can resolve disputes
    but not modify protocol state.
  </Card>
  <Card title="Protocol fee mechanism" icon="coins">
    `protocol_fee_bps = 20` (0.2%) and `lp_fee_bps = 30` (0.3%),
    capped at 100 bps each. Fee deduction happens on
    `AdManager.unlock`: bridger gets 99.5% of the destination
    amount, 0.3% stays in the maker's `ad.balance` as LP profit,
    0.2% routes to `protocol_fee_pool` (admin-set address).
  </Card>
  <Card title="Per-route bond config" icon="sliders">
    `BondConfig { bond_token, min_bond, bond_bps }` per route.
    For USDC ↔ USDC: bond in USDC, floor of 1 USDC, ratio of 50
    bps (0.5%). Same-token denomination keeps the math
    unit-clean without an oracle. Future cross-token bonds get
    oracle integration as a separate enhancement.
  </Card>
  <Card title="Reconciliation listener" icon="binoculars">
    Off-chain HA service, two-region. Watches both chains via
    private RPC providers, detects settlement discrepancies
    within a 5-minute SLA, alerts on-call. Read-only — no
    contract authority.
  </Card>
</CardGroup>

## How the protocol fee works

The fee structure mirrors the pitch deck's revenue model: every
bridge transaction incurs a flat 0.5% total — 0.3% to the LP
(the maker), 0.2% to the protocol.

Fees are deducted **destination-side** at `unlock`, from the
maker's locked amount:

| Party | Source side | Destination side | Net per trade (1:1 cross-chain) |
|---|---|---|---|
| Bridger | Deposits X | Receives 0.995 × X | -0.5% × X |
| Maker | Receives X | Locked balance reduces by 0.997 × X | +0.3% × X |
| Protocol | — | +0.002 × X to fee pool | +0.2% × X |

The bridger's net cost is 0.5% of the bridged amount. The maker
earns 0.3% of bridge volume. The protocol takes 0.2%.

Bps values are admin-configurable but timelocked, and capped at
100 bps each in the contract to bound future admin-key abuse.

## The launch route — USDC ↔ USDC

ProofBridge does **not** issue tokens. T3 launches with one
production route between two existing token contracts:

| Side | Asset | Issuer |
|---|---|---|
| Ethereum mainnet | USDC (ERC-20) | Circle |
| Stellar mainnet | USDC (Stellar Asset Contract) | Circle |

ProofBridge configures the route via `set_token_route` on each
chain (timelocked); Circle's contracts are unmodified.

Future routes — including any native-token bridging (XLM, ETH)
— wait for ecosystem partners to issue equivalent assets on the
counterpart chain. Adding a route post-launch is config-only:
`set_token_route` + `set_settlement_window` + `set_bond_config`,
all timelocked admin calls.

## Mainnet deployment is audit-gated

The SCF Audit Bank engagement runs during T3 and **gates**
mainnet deployment:

<Steps>
  <Step title="T2 contract code freezes">
    End of Phase 3 / Tranche 2. T3 contract changes (timelock
    module, fee mechanism, per-route bond config) are staged in
    the repo but not yet deployed to mainnet.
  </Step>
  <Step title="Audit kickoff">
    Auditors review the full T1 + T2 + T3 contract suite.
  </Step>
  <Step title="Wave 1 findings closed">
    Critical / high findings remediated. Audit Bank confirms
    closure.
  </Step>
  <Step title="Mainnet deployment begins">
    Deploy contracts, run initial config, activate timelock,
    hand off admin to multisig.
  </Step>
  <Step title="Production routes verified">
    First-tx ceremony for the USDC route in both directions.
    Tx hashes recorded on `/reference/smart-contracts`.
  </Step>
  <Step title="Public announcement">
    Frontend + SDK released, public docs go live.
  </Step>
  <Step title="4-week post-launch hardening">
    Daily uptime reports, bug fixes, audit medium/low
    remediation, integration partner support, perf tuning.
  </Step>
</Steps>

If audit timing slips, mainnet deploy slips. We don't ship a
known-broken launch.

## Trust model at mainnet

| Assumption | Strength | Defense |
|---|---|---|
| Deposits redeemable only via valid proof | Cryptographic | UltraHonk + nullifier + EIP-712 binding (Phase 1) |
| Counterparties consent before settlement | Cryptographic | BLS aggregate gates `unlock` (Phase 2) |
| Existing ad routes are immune to admin compromise | Contract invariant | Route commitment per ad (Phase 2) |
| Rogue admin can't move funds in < 24h | Operational | Timelock + multisig (Phase 4) |
| Disputes resolve fairly | Bounded human | Multisig arbiter, ≥ 2 non-team, public outcome trail (Phase 4) |

The arbiter remains the largest residual trust assumption. Phase 5
addresses it by handing the role to DAO governance.
